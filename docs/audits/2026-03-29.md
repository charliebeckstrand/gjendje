All prior audit findings (9 files) are resolved. This audit covers new findings discovered during a full codebase review.

## Finding 1 — Bucket adapter double notification outside `batch()` (HIGH)

**File:** `src/adapters/bucket.ts` — lines 127–134, 260–266

**Description:**
When `set()` is called on a bucket adapter **outside** of `batch()`, listeners receive **two** notifications for a single value change.

**Root cause:**
The bucket's `set()` method calls `delegate.set(value)`, which triggers the storage adapter's notification. That notification propagates through the bucket's delegate subscription (line 130–133), which calls `notify(notifyListeners)` — firing bucket listeners immediately. Then bucket's own `set()` method **also** calls `notify(notifyListeners)` (line 264–265), firing the same listeners a second time.

Inside `batch()`, the deduplication in `notify()` prevents the second enqueue. Outside `batch()`, both fire immediately.

**Impact:** Listeners see two identical updates per `set()`. Effects, UI bindings, and devtools subscriptions fire twice, wasting CPU and potentially causing glitches.

**Fix:** Remove the redundant `notify(notifyListeners)` from the bucket `set()` method and let the delegate subscription handle all notifications consistently.

- [x] PATCHED

## Finding 2 — `createOptimizedListeners` counter desync on duplicate subscriptions (LOW)

**File:** `src/listeners.ts` — lines 134–151

**Description:**
`createOptimizedListeners` maintains a separate `listenerCount` counter alongside the `listenerSet` (a `Set`). When the **same** listener function reference is subscribed twice, `listenerSet.add()` is a no-op (Set deduplication) but `listenerCount++` still increments. This desyncs the counter from the actual Set size.

**Impact:** The `singleListener` fast-path optimization may not activate when it should (minor performance regression). No correctness impact — the snapshot fallback path always iterates the real Set.

**Fix:** Use `listenerSet.size` to derive state instead of maintaining a separate counter.

- [x] PATCHED

## Finding 3 — `computed`/`select` use `===` instead of `Object.is` for notification skip (LOW)

**Files:** `src/computed.ts` — line 121, `src/select.ts` — line 105

**Description:**
The diamond-dedup check (`if (value === prev) return`) uses strict equality instead of `Object.is`. This has two edge cases:
- `NaN`: `NaN === NaN` is `false`, so a computed returning `NaN` triggers spurious notifications on every recompute.
- `+0` / `-0`: `+0 === -0` is `true`, so a transition from `+0` to `-0` is silently swallowed.

**Impact:** Spurious notifications for `NaN`-valued computeds; missed notifications for ±0 transitions. Both are rare in practice.

**Fix:** Replace `===` with `Object.is` to match the rest of the codebase (watchers, state equality checks).

- [x] PATCHED

## Finding 4 — Vue `useGjendje` re-runs selector on every `.value` access (LOW)

**File:** `src/vue/index.ts` — line 45

**Description:**
The `customRef` getter always calls `selector(instance.get())` on every `.value` access, even when the underlying value hasn't changed. The `last` variable (updated via subscription) already tracks the latest selected value but is not used by the getter.

**Impact:** Unnecessary selector re-execution on every template re-read. Minor performance issue for expensive selectors.

**Fix:** Return `last` from the getter when a selector is present, since `last` is kept in sync by the subscription callback.

- [x] PATCHED

## Finding 5 — `afterHydration()` doesn't catch callback errors (MEDIUM)

**File:** `src/ssr.ts` — lines 31–39

**Description:**
The `afterHydration()` utility calls `fn()` then `resolve()` sequentially inside `requestAnimationFrame` / `setTimeout`. If `fn()` throws, `resolve()` is never reached, leaving the returned promise pending forever. This promise is assigned to `instance._s.hydrated`, so `await instance.hydrated` would hang indefinitely.

**Current mitigation:** The hydration callback in `core.ts` (lines 1064–1101) has its own try/catch, so `fn()` doesn't throw in practice today. But `afterHydration` is a general utility and should be defensively safe.

**Impact:** A future refactor that removes the callback's try/catch — or a new caller of `afterHydration` — would silently hang the hydrated promise.

**Fix:** Wrap `fn()` in try/catch inside `afterHydration` and always call `resolve()`.

- [x] PATCHED

## Finding 7 — `configure()` silently accepts unknown keys (LOW)

**File:** `src/config.ts` — line 216

**Description:**
`configure()` iterates `Object.keys(config)` and merges every key into `globalConfig` without checking if the key is a valid `GjendjeConfig` property. A typo like `configure({ logLvel: 'warn' })` is silently accepted, leaving the user with no warning that their config has no effect.

**Impact:** Silent misconfiguration. Typos are not caught.

**Fix:** Add a `KNOWN_CONFIG_KEYS` set and log a warning when an unknown key is encountered.

- [x] PATCHED

## Finding 8 — `readonly()` doesn't shadow `onChange` (LOW)

**File:** `src/readonly.ts` — line 28

**Description:**
The `readonly()` wrapper shadows `set`, `reset`, `patch`, and `intercept`, but does not shadow `onChange`. While `onChange` is a subscription-like method, it registers a handler on the underlying writable instance. In untyped JS, this allows callers with a "read-only" reference to register change handlers that receive both new and previous values — a capability that should be restricted to writable consumers.

**Impact:** Minor API inconsistency. TypeScript hides `onChange` from `ReadonlyInstance<T>`, but JS callers can bypass this.

**Fix:** Add `onChange: { value: undefined }` to the property descriptor.

- [x] PATCHED

## Finding 9 — `previous()` uses `!==` instead of `Object.is` for notification skip (LOW)

**File:** `src/previous.ts` — line 77

**Description:**
The `previous()` function uses `old !== prev` to decide whether to notify subscribers. This has the same NaN/±0 edge cases as Finding 3 (computed/select): `NaN !== NaN` is `true`, causing spurious notifications every time the previous value is NaN.

**Impact:** Spurious notifications for NaN-valued previous instances; missed notifications for ±0 transitions.

**Fix:** Replace `old !== prev` with `!Object.is(old, prev)`.

- [x] PATCHED

## Finding 10 — `MemoryStateImpl.reset()` callback order inconsistency (LOW)

**File:** `src/core.ts` — lines 764–787

**Description:**
In `StateImpl.reset()`, `config.onReset` fires **before** instance `changeHandlers` (via `_notifyChange()`). In `MemoryStateImpl.reset()`, the inlined code fires `changeHandlers` **before** `config.onReset` — a behavioral inconsistency between memory and non-memory scopes.

**Impact:** Observable callback ordering difference between scopes. Code relying on `onReset` firing before `onChange` handlers would behave differently for memory-scoped state.

**Fix:** Reorder `MemoryStateImpl.reset()` to fire `config.onReset` before `changeHandlers`, matching `StateImpl`.

- [x] PATCHED

## Finding 11 — Storage/URL adapter `persist` notification mismatch (HIGH)

**Files:** `src/adapters/storage.ts` — line 215, `src/adapters/url.ts` — line 135

**Description:**
When `persist` is set (partial key persistence), the adapter's `set()` method notifies listeners with the raw input value instead of the merged value that `get()` returns. After `set({a: 3, b: 4})` with `persist: ['a']`, `get()` correctly returns `{a: 3, b: 2}` (merged with defaults), but subscribers were notified with `{a: 3, b: 4}` (unmerged).

**Impact:** Subscribers see a different value than `get()` returns, violating the fundamental contract that notification values match the current state. Effects, UI bindings, and devtools subscriptions all receive incorrect values.

**Fix:** Use the post-`write()` `cachedValue` (which is already correctly merged) as the notification value instead of the raw input.

- [x] PATCHED

## Finding 12 — Registry `onRegister` skipped on re-registration of destroyed instance (TRIVIAL)

**Files:** `src/registry.ts` — lines 63, 31

**Description:**
Both `registerNew` and `registerByKey` replace a destroyed instance in the registry without calling the `config.onRegister` callback. In practice this path is rarely reached because `destroy()` calls `unregisterByKey()` first, but if an adapter's `destroy()` throws before unregistering, the new instance silently registers without firing the callback.

**Impact:** Defensive correctness. The existing test "fires again after destroy + re-create" passes because normal destroy always unregisters first.

**Fix:** Call `safeCallConfig(config.onRegister, ...)` in both re-registration paths.

- [x] PATCHED

## Finding 13 — `withHistory` undo/redo on destroyed instance corrupts stacks (LOW)

**File:** `src/enhancers/history.ts` — line 72 (`navigate()`)

**Description:**
When the underlying instance is destroyed directly (not via the `withHistory` wrapper's `destroy()`), calling `undo()` or `redo()` still mutates the past/future stacks even though `instance.set()` is a no-op on destroyed instances. This silently corrupts the history stacks.

**Impact:** Silent history corruption if the underlying instance is destroyed independently. Low severity because users shouldn't be calling undo/redo after destroying the instance, and the wrapper's own `destroy()` clears both stacks.

**Fix:** Add `instance.isDestroyed` guard to the `navigate()` early return.

- [x] PATCHED
